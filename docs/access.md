# Access and configuration management

## Cluster admin access (humans)
- The kubeconfig `/etc/kubernetes/admin.conf` lives on the VM only; it's pulled to a local `./kubeconfig` (chmod 600) by `bootstrap/03-fetch-kubeconfig.sh`.
- The kubeconfig presents a client cert generated by Kubespray's PKI — it's not a static long-lived password.
- For any additional admin a per-user CSR + RoleBinding pattern is the next step — see `security.md`.

## GitOps as the source of truth
- All cluster state (workloads, ingress, monitoring, business app) is **declared in this git repo** and applied by **ArgoCD**.
- A single root `Application` (`argocd/root-app.yaml`) walks `argocd/apps/` and creates a child `Application` per concern. This is the **app-of-apps** pattern, identical to lab9/lab10.
- `syncPolicy.automated.{prune,selfHeal}: true` means ArgoCD will revert any manual `kubectl edit` within seconds — git is the only legitimate way to change cluster state.

## Sync ordering (sync-wave annotations)
```
wave 0 → sealed-secrets controller (provides SealedSecret CRD)
wave 1 → local-path-provisioner (provides StorageClass)
wave 2 → ingress-nginx
wave 3 → kube-prometheus-stack, loki
wave 5 → grafana dashboards (after monitoring is up)
wave 10 → business chart (depends on storage + sealed-secrets)
```

## Secrets
- Plaintext secrets never enter git. We use **Bitnami sealed-secrets**: the controller's public key encrypts a `Secret` into a `SealedSecret` (which IS safe to commit).
- The controller running in `kube-system` is the only thing that can decrypt — even an attacker reading the entire repo learns nothing.

## Live verification
```bash
kubectl -n argocd get app -o wide              # all Synced/Healthy
kubectl -n kube-system get deploy sealed-secrets
ls -la kubeconfig                              # should be 0600
git log --oneline | head                       # one commit per intentional change
```
